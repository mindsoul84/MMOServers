1-1. 락 설계 잠재적 위험 제거
---------------------------------------------------------------------
  수정 파일: GameServer/GameServer.h, GatewayHandlers.cpp,
             WorldHandlers.cpp, MonsterManager.cpp

  [문제]
  playerMutex_와 monsterMutex_ 두 개의 shared_mutex를 사용하면서
  락 획득 순서 규칙(monsterMutex_ -> playerMutex_)을 수동으로 관리.
  GatewayHandlers의 공격 처리에서 playerMutex_ -> monsterMutex_ 순서로
  획득하는 경로가 존재하여, 현재는 shared_lock끼리라 데드락이 발생하지
  않지만 향후 unique_lock이 필요한 로직 추가 시 데드락 위험이 잠재.

  [수정]
  game_strand_ 도입으로 playerMutex_, monsterMutex_ 자체를 제거.
  strand가 모든 게임 로직을 직렬화하므로 락 순서 규칙 자체가 불필요.
  데드락 가능성이 원천적으로 0이 됨.

1-2. Monster::Update() 스레드 안전성 확보
---------------------------------------------------------------------
  수정 파일: GameServer/Monster/Monster.h, Monster.cpp, MonsterManager.cpp

  [문제]
  Monster의 Update(), UpdateChase(), UpdateAttack() 등이 position_,
  state_, path_index_ 같은 멤버를 mtx_ 없이 직접 읽고 씀.
  io_context.run()을 여러 워커 스레드에서 호출하고 있어 타이머 콜백이
  어느 스레드에서 실행될지 보장 없음. Race Condition 위험 잠재.

  Monster::CalculatePath()의 결과 콜백이 io_context에 post되어
  AI Tick과 다른 스레드에서 몬스터 상태를 변경할 수 있었음.

  [수정]
  (1) ScheduleNextAITick 타이머 콜백을 bind_executor(game_strand_)로 바인딩
      -> Monster::Update()가 항상 game_strand_에서 실행됨
      -> 패킷 핸들러와 동일 스레드에서 몬스터 상태 접근
      -> 몬스터 개별 뮤텍스(Monster::mtx_) 제거 가능

  (2) CalculatePath 결과 콜백을 io_context 대신 game_strand_에 post
      -> 경로 적용이 AI Tick, 핸들러와 동일 strand에서 실행
      -> current_path_, path_index_ 변경 시 Race Condition 제거

  (3) Monster::mtx_ 제거
      -> TakeDamage(), Die(), GetHp(), SetHp()에서 lock_guard 제거
      -> game_strand_가 직렬화를 보장하므로 개별 뮤텍스 불필요

1-3. 에러 핸들링 및 연결 관리 보완
---------------------------------------------------------------------

  [문제]

  OnDisconnected 시 game_strand_에 유저 일괄 정리를 post.
  상세 내용은 아래 2-2절 참조.

2. 코드 개선 (리팩토링)

2-1. 스레드 모델 근본적 재설계 (game_strand_ 도입)
---------------------------------------------------------------------
  수정 파일: GameServer/GameServer.h, GatewaySession.cpp,
             WorldConnection.cpp, GatewayHandlers.cpp,
             WorldHandlers.cpp, MonsterManager.cpp, Monster.h, Monster.cpp

  [설계 원칙]
  "게임 로직은 싱글 스레드(strand), 네트워크 I/O는 멀티 스레드"

  이 원칙은 실무 MMORPG 서버에서 가장 보편적인 패턴입니다.
  동시성 버그의 대부분이 설계 단계에서 제거됩니다.

  [구현 상세]

  (1) GameContext에 game_strand_ 추가
      boost::asio::io_context::strand game_strand_(io_context);
      -> io_context의 워커 스레드 중 하나에서 실행되지만
         동시에 두 개 이상의 콜백이 실행되지 않음을 보장

  (2) GatewaySession::ReadHeader/ReadPayload 수정
      -> 패킷 수신 후 Dispatch를 game_strand_에 post
      -> 페이로드는 복사본(shared_ptr<vector<char>>)으로 전달
         (ReadHeader가 즉시 다음 패킷을 읽기 시작하므로
          payload_buf_가 덮어쓰이기 전에 복사 필요)
      -> 네트워크 읽기는 I/O 스레드에서 계속 병렬 처리

  (3) WorldConnection::ReadHeader/ReadPayload 수정
      -> GatewaySession과 동일한 패턴으로 game_strand_ 디스패치

  (4) ScheduleNextAITick 수정
      -> bind_executor(game_strand_)로 타이머 콜백 바인딩
      -> AI Tick이 패킷 핸들러와 동일 strand에서 실행

  (5) Monster::CalculatePath 수정
      -> 결과 콜백을 io_context 대신 game_strand_에 post

  [제거된 동기화 프리미티브]
  - GameContext::playerMutex_  (UTILITY::Lock / shared_timed_mutex) -> 제거
  - GameContext::monsterMutex_ (UTILITY::Lock / shared_timed_mutex) -> 제거
  - PlayerInfo::mtx            (std::mutex)                         -> 제거
  - Monster::mtx_              (std::mutex)                         -> 제거

  [유지되는 동기화 프리미티브]
  - GameContext::gatewaySessionMutex (std::mutex)
    -> 세션 라이프사이클 관리용, 게임 상태와 독립
    -> accept 콜백(I/O 스레드)과 OnDisconnected에서 접근
  - Zone 내부 Sector::mutex_ (UTILITY::Lock)
    -> Zone 자체의 방어적 동기화 (재사용 가능한 컴포넌트)
    -> 현재 game_strand_에서만 접근하므로 사실상 경합 없음
  - 각 세션의 Send용 strand_ (네트워크 I/O 직렬화)

2-2. 게이트웨이 장애 시 유저 일괄 정리 (Fault Tolerance)
---------------------------------------------------------------------
  수정 파일: GameServer/GameServer.h, GatewaySession.cpp,
             GatewayHandlers.cpp

  [문제]
  GatewaySession이 끊어졌을 때 해당 게이트웨이를 통해 접속한
  유저들의 정리가 수행되지 않음. playerMap에 좀비 유저가 잔존하여:
    - Zone AOI에서 유령 유저 발견
    - 브로드캐스트 시 끊어진 게이트웨이로 패킷 전송 시도
    - 메모리 누수 (PlayerInfo 미해제)

  [수정]
  (1) GameContext에 gatewayPlayerMap_ 추가
      -> Key: GatewaySession 원시 포인터
      -> Value: 해당 게이트웨이를 통해 접속한 유저의 account_id 집합
      -> game_strand_ 안에서만 접근하므로 별도 뮤텍스 불필요

  (2) GameContext에 유틸리티 메서드 추가
      - RegisterPlayerToGateway(gw, account_id): 유저 등록
      - UnregisterPlayerFromGateway(gw, account_id): 유저 해제
      - GetPlayersOfGateway(gw): 소속 유저 목록 반환
      - RemoveGatewayMapping(gw): 게이트웨이 매핑 전체 삭제

  (3) Handle_GatewayGameMoveReq에서 신규 유저 진입 시 RegisterPlayerToGateway 호출
  (4) Handle_GatewayGameLeaveReq에서 유저 퇴장 시 UnregisterPlayerFromGateway 호출

  (5) GatewaySession::OnDisconnected 전면 재작성
      -> game_strand_에 유저 일괄 정리 요청을 post
      -> GetPlayersOfGateway로 소속 유저 목록 조회
      -> playerMap, uidToAccount, Zone, Redis에서 일괄 삭제
      -> 세션 라이프사이클(gatewaySessions)은 별도 뮤텍스로 즉시 정리

  [장애 복구 시나리오]
  1. GatewayServer 프로세스가 비정상 종료됨
  2. TCP 연결이 끊어져 GatewaySession::ReadHeader 에러 발생
  3. OnDisconnected() 호출
  4. gatewaySessions에서 세션 즉시 제거 (더 이상 브로드캐스트 안 함)
  5. game_strand_에 유저 정리 작업이 예약됨
  6. strand에서 해당 게이트웨이의 모든 유저를 순회하며:
     - playerMap에서 삭제
     - uidToAccount에서 삭제
     - Zone에서 LeaveZone 호출
     - Redis에서 RemovePlayer 호출
  7. gatewayPlayerMap_에서 해당 게이트웨이 항목 삭제
  8. 로그 출력: "Gateway 장애 복구: N명의 유저 일괄 정리 완료"
