1-1. LoginServer Session::Send() strand 미적용 수정
---------------------------------------------------------------------
수정 파일: LoginServer/Session/Session.h, LoginServer/Session/Session.cpp

  [문제]
  GatewaySession, ClientSession, GameConnection, WorldConnection, ServerSession은
  모두 strand + send_queue 패턴이 적용되어 있었으나,
  LoginServer의 Session만 async_write를 strand 없이 직접 호출하고 있었음.

  DB 스레드에서 post된 콜백(Handle_LoginReq)이 login_res를 보내는 동시에
  다른 스레드에서 같은 세션에 패킷을 보낼 경우 concurrent async_write 발생.

  [수정]
  Session에 strand_ + send_queue_ + DoWrite() 패턴을 적용.
  프로젝트 내 모든 네트워크 세션 클래스가 동일한 동시성 보호를 갖추도록 통일.

  적용 완료된 전체 세션 목록 (6/6):
    GameServer/GatewaySession      (v1.0.3 적용)
    GatewayServer/ClientSession    (v1.0.3 적용)
    GatewayServer/GameConnection   (v1.0.3 적용)
    LoginServer/WorldConnection    (v1.0.3 적용)
    GameServer/WorldConnection     (v1.0.3 적용)
    WorldServer/ServerSession      (v1.0.7 적용)
    LoginServer/Session            (이번 수정에서 적용)

    변경 후: SELECT account_uid, password_hash, salt FROM Accounts WHERE account_name = ?

      변경 전: INSERT INTO Accounts (account_id, password_hash, salt, input) VALUES (?, ?, ?, ?)
      변경 후: INSERT INTO Accounts (account_uid, account_name, password_hash, salt, input) VALUES (?, ?, ?, ?, ?)

    - 1번 파라미터: account_uid (SQL_C_SBIGINT / SQL_BIGINT 타입)
    - 2번 파라미터: account_name (기존 account_id에서 컬럼명 변경)
    - 3~5번 파라미터: password_hash, salt, input (동일)
    - 신규 가입 성공 시 생성된 account_uid를 out_account_uid로 반환

1-7. TokenNotify 브로드캐스트 범위 오류 + DB INSERT 동시성 충돌 수정
---------------------------------------------------------------------
수정 파일: LoginServer/LoginServer.cpp, Common/DB/DBManager.cpp

  [현상]
  LoginServer 콘솔에 다음 에러 2가지가 동시 발생:
    (1) "등록되지 않은 패킷 ID 수신: 1030 (payloadSize=90) - 무시됨"
    (2) "[ERROR] [DBManager] INSERT Execute 실패"

  [원인 분석 - 에러 1: 미등록 패킷 ID 1030]
  WorldServer의 g_serverSessions에는 GameServer와 LoginServer가 모두 포함됨.
  1-2에서 추가한 TokenNotify 브로드캐스트가 g_serverSessions 전체를 순회하므로
  LoginServer에도 PKT_WORLD_GAME_TOKEN_NOTIFY(1030) 패킷이 전달됨.
  LoginServer에는 1030 핸들러가 등록되어 있지 않아 에러 로그 출력.

  [수정 - 에러 1]
  LoginServer의 worldDispatcher에 빈 핸들러(no-op)를 등록하여
  패킷을 수신만 하고 무시하도록 함. LoginServer는 토큰 통지를 처리할 필요가 없으므로
  의도적 무시가 올바른 설계.

    ctx.worldDispatcher.RegisterHandler(Protocol::PKT_WORLD_GAME_TOKEN_NOTIFY,
        [](std::shared_ptr<WorldConnection>&, char*, uint16_t) {
            // LoginServer에서는 토큰 통지를 처리하지 않음 (GameServer 전용)
        });

  [원인 분석 - 에러 2: INSERT Execute 실패]
  1-4에서 도입한 MAX(account_uid) + 1 방식의 UID 생성이 DB 스레드 간 경합.
  LoginServer의 db_thread_count=2인 상태에서 동시에 신규 가입이 발생하면:

    DB Thread 0: SELECT MAX(account_uid) -> 결과: 1500
    DB Thread 1: SELECT MAX(account_uid) -> 결과: 1500  (아직 Thread 0이 INSERT 전)
    DB Thread 0: INSERT ... account_uid=1501 -> 성공
    DB Thread 1: INSERT ... account_uid=1501 -> PK 중복! INSERT Execute 실패

  Autocommit 모드에서는 SELECT와 INSERT가 별도 암묵적 트랜잭션으로 실행되어
  두 스레드가 같은 MAX 값을 읽을 수 있음.

  [수정 - 에러 2]
  ODBC 수동 트랜잭션으로 MAX 조회와 INSERT를 하나의 원자적 단위로 묶음:

    1. SQLSetConnectAttr(SQL_AUTOCOMMIT_OFF)로 수동 트랜잭션 시작
    2. SELECT MAX(account_uid) WITH (UPDLOCK, HOLDLOCK) 실행
       -> UPDLOCK: 업데이트 잠금 획득 (다른 트랜잭션의 동시 읽기를 대기시킴)
       -> HOLDLOCK: 트랜잭션 완료까지 잠금 유지 (SERIALIZABLE과 동일 효과)
    3. INSERT 실행
    4. SQLEndTran(SQL_COMMIT)으로 커밋 (또는 실패 시 SQL_ROLLBACK)
    5. SQLSetConnectAttr(SQL_AUTOCOMMIT_ON)으로 Autocommit 복원

  모든 에러 경로에서 ROLLBACK + Autocommit 복원을 보장하여
  ODBC 연결 상태가 오염되지 않도록 처리.

  효과:
    - Thread 0이 MAX 조회 시 UPDLOCK을 획득하면
      Thread 1의 MAX 조회는 Thread 0의 COMMIT까지 대기
    - Thread 0이 COMMIT 후 Thread 1이 MAX를 읽으면 1501이 아닌 1502를 얻음
    - PK 충돌이 원천적으로 발생하지 않음

2-1. MonsterManager::ScheduleNextAITick() 함수 분리
---------------------------------------------------------------------
수정 파일: GameServer/Monster/MonsterManager.h,
           GameServer/Monster/MonsterManager.cpp

  [문제]
  IDLE/CHASE/ATTACK/DEAD 처리, 리스폰 로직, AOI 브로드캐스트까지
  약 200줄이 하나의 타이머 콜백 람다에 들어있었음.
  이 함수 내부에서 monsterMutex_ shared_lock을 잡은 채로
  playerMutex_ shared_lock을 중첩 획득하는데,
  이 순서가 다른 경로와 일치하는지 코드를 추적하기 극히 어려웠음.

  [수정]
  상태별 처리를 독립 함수로 분리:

    ProcessDeadMonster()    - 사망 상태 (리스폰 타이머, 부활 알림)
    ProcessIdleMonster()    - IDLE 상태 (어그로 탐지)
    ProcessChaseMonster()   - CHASE 상태 (타겟 추적/포기)
    ProcessAttackMonster()  - ATTACK 상태 (타겟 유지/포기)
    SyncMonsterPosition()   - 이동 후 Zone 갱신 및 네트워크 동기화

  각 함수에 락 획득 순서를 주석으로 명시:
    "락 순서: monsterMutex_(이미 보유) -> playerMutex_ (OK)"
    이 순서는 GatewayHandlers.cpp의 공격 처리와 일관됨.

  ScheduleNextAITick은 switch(state)로 상태별 함수를 호출하는
  간결한 디스패치 루프로 축소됨 (약 40줄).

  효과:
    - 각 함수의 락 범위가 명확하여 데드락 추적 용이
    - 상태별 단위 테스트 가능
    - 하드코딩된 매직 넘버를 GameConstants 상수로 교체
      (1.0f -> AGGRO_RANGE, 3.0f -> CHASE_RANGE, 0.05f -> POSITION_EPSILON)
